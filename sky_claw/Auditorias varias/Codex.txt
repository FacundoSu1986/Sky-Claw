Aplicación de reglas
Regla 1 — SRE primero: priorizo SLOs, rollback, observabilidad, control de error budget y recuperación sobre velocidad de entrega.
Regla 2 — Seguridad antes que DevOps/QA: no se considera production-ready si secretos, sandboxing, SSRF, path traversal o ejecución de herramientas externas no están validados.
Regla 3 — Testing shift-left: los escenarios de fallo deben automatizarse antes del despliegue; no basta con pruebas manuales.
Regla 4 — Observabilidad obligatoria: logs JSON, correlation IDs, métricas, trazas y alertas deben ser accionables antes del go-live.
Excepciones / Trade-offs: no ejecuté una batería completa porque el entorno local no tiene dependencias instaladas; el comando de pytest falló por ausencia de pydantic, lo que documento como limitación ambiental y como bloqueo de readiness si se reproduce en CI.

6. Estrategia de Rollback
6.1 Rollback técnico de una operación
Sky-Claw ya declara mecanismos de rollback automático, snapshots, journal, locks y cleanup de procesos externos. DEPLOYMENT.md (líneas 189-199) Para producción, el runbook operativo debe formalizar:
Antes de cada mutación
Crear snapshot de archivos afectados.
Registrar transaction_id en journal.
Adquirir lock con TTL y heartbeat.
Validar espacio en disco.
Validar que no haya procesos MO2/xEdit/LOOT conflictivos abiertos.

Durante la operación
Asociar todos los logs al mismo correlation_id.
Registrar started/completed/failed.
Aplicar timeout por herramienta externa.
Mantener heartbeat del lock.

Si falla
Detener subprocess.
Ejecutar rollback desde snapshots en orden inverso.
Marcar transacción rolled_back.
Emitir alerta P0/P1 si el rollback falla o tarda más del SLO.

Después
Verificar checksums/rutas.
Verificar journal sin pending.
Revisar logs por ERROR.
Generar reporte para post-mortem blameless.

6.2 Rollback de release
Escenario A — Falla durante despliegue del .exe
Detener distribución del artefacto.
Revocar release/tag si aún no fue publicado públicamente.
Marcar build como fallido en CI/CD.
Conservar artefactos/logs/SBOM para análisis.
Restaurar versión anterior firmada.
Publicar nota interna de incidente.
Escenario B — Falla en ejecución local de usuario
Pedir al operador no cerrar por kill -9 salvo último recurso; el runbook indica que SIGTERM debe permitir cleanup. DEPLOYMENT.md (líneas 197-198)
Exportar:logs/sky_claw.log
logs/watcher.log
logs/watcher_security.log
journal SQLite
correlation_id

Ejecutar recuperación:rollback automático si aplica;
restore snapshot manual vía API si el rollback automático no corrió;
restaurar backup externo del perfil MO2 si los snapshots no bastan.

Deshabilitar temporalmente herramientas externas afectadas.
Crear post-mortem con causa raíz, blast radius, fix y prueba de regresión.
Escenario C — Fuga de secreto
Rotar inmediatamente token LLM/Nexus/Telegram/WS.
Invalidar sesiones o token files WS.
Revisar logs históricos.
Ejecutar secret scanning.
Abrir incidente P0.
Añadir patrón de redacción faltante y prueba de regresión.
7. Recomendación final de Go/No-Go
No-Go hasta cumplir estos bloqueantes
E2E real documentado en VM Windows/MO2 descartable.
Pipeline de release con tag, .exe firmado y smoke test.
CI verde con dependencias bloqueadas.
Pruebas de rollback con fallo inyectado en cada etapa crítica.
Pruebas de redacción de secretos en mensajes, args, extra y tracebacks.
trace_id emitido en JSON o decisión explícita de no usar tracing distribuido.
Cobertura del handler de excepciones async en GUI o prueba equivalente bajo NiceGUI/uvicorn.
Alertas mínimas para rollback failed, journal pending, secret leakage y subprocess timeout.
Go limitado aceptable
Puede autorizarse un pilot controlado si:
corre en VM o perfil MO2 descartable;
no se opera sobre instalación Skyrim principal;
se ejecuta solo con backup externo previo;
el operador revisa logs y journal tras cada operación;
todas las mutaciones requieren HITL;
hay plan manual de restauración.
